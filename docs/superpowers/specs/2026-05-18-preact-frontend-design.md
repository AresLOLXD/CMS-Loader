# Spec: Preact Frontend Rewrite

**Date:** 2026-05-18
**Status:** Approved

## Overview

Replace all legacy frontend (static HTML, vanilla JS, EJS views) with a Preact SPA built with Vite. The Express 5 backend remains unchanged except for one new endpoint. Add automated semantic versioning via Husky + semantic-release. Update the README to reflect the new stack.

## Goals

- Single Preact SPA that covers login, CSV upload, column mapping, and job progress
- Automated semver versioning tied to Conventional Commits
- App version visible in the UI, read from `package.json`
- No changes to any existing Express controller, middleware, or session logic

## Non-Goals

- Replacing Express (evaluated and rejected — see brainstorming notes)
- Server-side rendering on the frontend
- Adding a database or changing the session store
- Migrating the test suite

### Integration with Express

**Development:** Two processes, one effective port for the browser.
- Express runs on `:3000`
- Vite dev server runs on `:5173`, proxying all non-asset requests to Express:
  - `/api`, `/login`, `/analyzeCSV`, `/registerUsers`, `/addParticipation`, `/jobs`

**Production:**
- `vite build` outputs to `frontend/dist/`
- Express serves `frontend/dist/` as static files via `express.static`
- Express serves `frontend/dist/index.html` as fallback for any non-API route (SPA catch-all)

**Root scripts updated:**
```json
"dev:backend":  "ts-node src/index.ts",
"dev:frontend": "vite --config frontend/vite.config.ts",
"build":        "rimraf dist/ && tsc && copyfiles ... && vite build --config frontend/vite.config.ts",
"release":      "semantic-release --no-ci"
```

Note: the existing `copy-files` step (EJS/static asset copy) is removed from `build` once the legacy frontend files are deleted.

## State Management

Global state lives in `frontend/src/signals.ts` using Preact Signals (`@preact/signals`).

| Signal | Type | Description |
|---|---|---|
| `authStatus` | `'checking' \| 'authenticated' \| 'unauthenticated'` | Session state on app mount |
| `csrfToken` | `string` | Fetched once after auth, reused across all mutating requests |
| `wizardStep` | `'upload' \| 'mapping' \| 'processing' \| 'done'` | Current wizard step |
| `mode` | `'users' \| 'contest'` | Operation mode selected by the user |
| `columns` | `string[]` | CSV column names returned by `/analyzeCSV` |
| `mapping` | `Record<string, string>` | Field → CSV column assignments from MappingStep |
| `jobId` | `string \| null` | Job ID returned by `/registerUsers` or `/addParticipation` |

## Data Flow

### App Mount

1. `GET /api/me` → `{ authenticated: boolean }`
   - `true` → set `authStatus = 'authenticated'`, fetch CSRF token
   - `false` → set `authStatus = 'unauthenticated'`
2. `GET /api/csrf-token` → `{ token: string }` → store in `csrfToken` signal

### Login

- `POST /login` with `{ username, password }` and `x-csrf-token` header
- Success → set `authStatus = 'authenticated'`
- Failure → show inline error

### UploadStep

- User selects mode (`users` or `contest`) and a `.csv` file
- `POST /analyzeCSV` with `FormData{ archivo }` and `x-csrf-token`
- Response: `{ success: true, data: { columnas: string[] } }`
- Store `columns`, advance `wizardStep` to `'mapping'`

### MappingStep

- Renders one `<select>` per CMS field (driven by `fieldsConfig[mode]`)
- Options populated from `columns` signal
- `POST /registerUsers` or `POST /addParticipation` with `JSON{ ...mapping }` and `x-csrf-token`
- Response: `{ jobId: string }`
- Store `jobId`, advance `wizardStep` to `'processing'`

### ProcessingStep

- Opens `new EventSource(\`/jobs/${jobId}/events\`)`
- `progress` event → update progress bar (`processed`, `total`, `percent`)
- `done` event → `GET /jobs/${jobId}/result` → download blob as CSV → advance to `'done'`
- `job-error` event → display error, allow retry from MappingStep
- `onerror` → display connection error, allow retry
- `EventSource` is closed in `useEffect` cleanup; if user returns to this step, a new `EventSource` is opened with the stored `jobId`

### CSRF Token Refresh

If any mutating request returns HTTP 403, the app re-fetches `GET /api/csrf-token` and retries the request once before showing an error.

```
Response 200: { authenticated: false }  otherwise
```

This is the only change to the Express backend.

## Automated Versioning

### Tools

| Tool | Purpose |
|---|---|
| `husky` | Git hooks runner |
| `@commitlint/cli` + `@commitlint/config-conventional` | Validates commit message format |
| `semantic-release` | Bumps version, generates CHANGELOG, creates git tag |
| `@semantic-release/changelog` | Writes `CHANGELOG.md` |
| `@semantic-release/git` | Commits updated `package.json` + `CHANGELOG.md` |

### Commit Message Format (Conventional Commits)

```
feat: add contest mode progress bar     → minor bump (0.x.0)
fix: handle empty CSV column list       → patch bump (0.0.x)
feat!: replace session store            → major bump (x.0.0)
chore: update dependencies              → no release
```

### Git Hooks

**`commit-msg`** — blocks the commit if the message does not follow Conventional Commits:
```sh
npx --no -- commitlint --edit "$1"
```

**`pre-push`** — runs semantic-release before pushing to `main`:
```sh
npx semantic-release --no-ci
```

Semantic-release pushes the version tag separately; this does not retrigger the pre-push hook because it is a tag push, not a branch push. If there are no releasable commits since the last tag (e.g. only `chore:` commits), semantic-release exits 0 without creating a release and the push proceeds normally.

### Release Flow

```
git commit -m "feat: add processing step progress ring"
  → commit-msg hook validates format ✓

git push origin main
  → pre-push hook runs semantic-release --no-ci
  → reads commits since last tag
  → bumps version in package.json (e.g. 1.0.0 → 1.1.0)
  → generates / updates CHANGELOG.md
  → commits both files with "chore(release): 1.1.0"
  → creates and pushes git tag v1.1.0
  → original push proceeds
```

### Version in UI

`vite.config.ts` reads `version` from the root `package.json` and exposes it as `import.meta.env.VITE_APP_VERSION`. The footer component renders `v{import.meta.env.VITE_APP_VERSION}`.
